# Claude Code Prompt — Benchmark Suite Generation

## Preamble

This document is the instruction set used to generate the benchmark suite for a comparative evaluation of open-source container vulnerability scanners, forming part of a BSc (Hons) Cyber Security Honours dissertation at Glasgow Caledonian University (2025-2026). The suite is generated by Claude; equivalent instruction sets are used with Gemini and Copilot to produce comparable datasets for cross-LLM analysis.

This file is retained as part of the project record for traceability and reproducibility purposes.

---

## Critical Rules — Read Before Starting

- **Never guess.** If a CVE score, affected version, or any other factual item cannot be confirmed from an official source, stop and say so. Do not proceed with assumed values.
- **CVSS scores must come from NIST NVD only** (`https://nvd.nist.gov`). Do not use Reddit, blogs, vendor advisories, or any non-canonical source for scores.
- **All output must be verifiable.** Every CVE selected must be reachable at `https://nvd.nist.gov/vuln/detail/{CVE-ID}`.
- **Do not build containers.** File generation only — no `docker build`, no `kaniko` invocation.
- **Do not modify any existing files** outside the `~/Claude/benchmark-suite/` directory tree.
- **Make all changes directly** without asking for confirmation, explaining reasoning, or performing unnecessary checks. Just implement.

---

## Output Location

All output is created under:

```
~/Claude/benchmark-suite/
```

This is the root of the test suite. All directories and files are created relative to this root.

## Step 1 — Create Test Run Directory

Create the following directory and make it the working root for all subsequent steps:

```
~/Claude/benchmark-suite/claude/
```

Create `~/Claude/benchmark-suite/claude/README.md` with the following content:

```markdown
# Claude Benchmark Test Run

This directory contains the benchmark suite test run generated by Claude (Anthropic) as part of a comparative evaluation of open-source container vulnerability scanners.

Generated for: BSc (Hons) Cyber Security Honours Dissertation, Glasgow Caledonian University, 2025-2026.

## Purpose

This test run is one of three equivalent runs generated by different LLMs (Claude, Gemini, Copilot) from an identical specification. Divergence between runs is academically significant and forms part of the comparative analysis.

## Structure

Each subdirectory corresponds to a programming language test set. Within each test set:
- `{language}-set.csv` — ground truth file for automation
- `README.md` — vulnerability descriptions and summary table
- `base-{image}/` — baseline control container(s)
- `{CVE-ID}/` — one directory per test case
```

---

## Step 2 — Language Scope

The following five languages must be processed, each as a separate test set. Process them in this order:

1. Python
2. Go
3. Java
4. JavaScript
5. C++

Languages were selected based on the TIOBE long-term top 10 as of February 2026, excluding languages not native to Linux (Visual Basic, Delphi, C#) and application-specific languages (SQL). C is excluded as C++ functionally replaces it for this purpose.

---

## Step 3 — For Each Language: Research Phase

Before generating any files for a language, carry out the following research:

### 3.1 CVE Selection

Search the NIST NVD for the 12 most serious vulnerabilities affecting the language — either its runtime or its commonly used libraries. The search scope is **March 2013 (Docker's first release) to the present date**.

Selection criteria in priority order:
- CVSS v3.1 score, descending (preferred)
- Fall back to CVSS v2 score only where v3.1 is unavailable
- Prefer Critical (9.0+) and High (7.0-8.9) vulnerabilities
- Medium (4.0-6.9) may only be used if fewer than 10 Critical/High vulnerabilities exist
- Never use Low severity vulnerabilities under any circumstances

If 12 vulnerabilities cannot be found, a minimum of 10 is acceptable but must be documented in the language README.md. If fewer than 10 can be found, stop and inform the user — do not proceed with that language.

For each selected CVE record:
- CVE ID (or named vulnerability if no CVE assigned)
- CVSS score and version (v3.1 or v2)
- Affected library, runtime or package name
- Affected version (the specific vulnerable version to use)
- Brief description of the vulnerability

### 3.2 Base Image Selection

Select a base image for the test set following these rules in priority order:
1. Alpine-based slim image strongly preferred (e.g. `python:3.11-alpine`, `golang:1.21-alpine`)
2. If Alpine is unsuitable for a specific test case, use the official language slim image (e.g. `python:3.12-slim`)
3. Never use `latest` — always pin to a specific version
4. Where possible, pin to a specific image digest for reproducibility
5. If more than one base image is required across the test set, each gets its own `base-{image}/` directory

## Step 4 — For Each Language: Documentation Phase

Generate documentation **before** generating any test case files.

### 4.1 README.md

Create `~/Claude/benchmark-suite/claude/{language}/README.md` containing:

- A section for each selected vulnerability with:
  - Vulnerability name and CVE ID
  - CVSS score and severity grade
  - Affected library/runtime and version
  - Description not exceeding two lines
- A summary table at the end with columns: CVE ID, CVSS Score, Severity, Affected Library, Affected Version, Base Image
- A note if fewer than 12 test cases are included, explaining why
- A note for any test cases requiring a non-standard base image, explaining the deviation

### 4.2 CSV Ground Truth File

Create `~/Claude/benchmark-suite/claude/{language}/{language}-set.csv` with the following fields:

```
cve_id,cvss_score,affected_library,affected_version,test_case_path,base_image
```

Rules:
- One row per test case
- `cve_id` — CVE ID or named vulnerability identifier
- `cvss_score` — numeric score from NIST NVD (e.g. `9.8`)
- `affected_library` — library, package or runtime name
- `affected_version` — exact pinned version used in the test case
- `test_case_path` — relative path from the test run root, e.g. `claude/python/CVE-2021-44228`
- `base_image` — full base image reference including tag, e.g. `python:3.11-alpine`
- Include a row for each baseline container with `cve_id` set to the directory name (e.g. `base-alpine3.19`) and `cvss_score` set to `0.0`
- No header quoting — plain CSV

## Step 5 — For Each Language: Baseline Container

Create a baseline control container for each base image in use.

Directory name: `base-{imagename}` where `{imagename}` is the image name and tag with `:` replaced by a `-`, e.g. `base-python3.11-alpine` or `base-alpine3.19`.

### Dockerfile specification

```dockerfile
FROM {base_image}
# Baseline container — no application vulnerabilities
# Used to establish base image CVE baseline for scanner result normalisation
COPY app.{ext} .
CMD ["{runtime}", "app.{ext}"]
```

### Application specification

The application prints the following to stdout and exits:

```
Language: {language}
Image: {base_image}
Severity: baseline
```

The application must be written in the language of the test set. It must have no dependencies beyond the language runtime — no manifest file required for the baseline container.

### Image tag

```
claude-{language}-base-{imagename}
```

For example: `claude-python-base-python3.11-alpine`

## Step 6 — For Each Language: Test Cases

For each of the 12 (or minimum 10) selected CVEs, create a test case directory and its contents.

### 6.1 Directory

```
~/Claude/benchmark-suite/claude/{language}/{CVE-ID}/
```

If the vulnerability has no CVE ID, use the common name (e.g. `Log4Shell`).

### 6.2 Files to Generate

Generate the following files for each test case:

#### Dockerfile

Requirements:
- `FROM` the pinned base image for this test set (or the specific base image if a non-standard one is required)
- `COPY` the manifest file into the container before installing dependencies
- Install dependencies from the manifest file using the language-standard package manager
- `COPY` the application source file
- Set a default `CMD` to run the application
- Must be compatible with both `docker build` and `kaniko` — no BuildKit-specific syntax, no `--mount` flags, no `RUN --mount`
- Target architecture: `linux/amd64` — add `--platform=linux/amd64` to the `FROM` line

#### Manifest File

Language-standard dependency manifest, pinned to the specific vulnerable version:

| Language | Manifest file |
|---|---|
| Python | `requirements.txt` |
| Go | `go.mod` (and `go.sum` if needed) |
| Java | `pom.xml` |
| JavaScript | `package.json` |
| C++ | `CMakeLists.txt` or `conanfile.txt` as appropriate |

All dependencies must be version-pinned. No floating versions (no `>=`, no `~=`, no `*`).

#### Application Source File

A minimal application in the language under test that:
- Imports or uses the vulnerable library at the pinned version
- Prints the following to stdout and exits cleanly:
  ```
  Language: {language}
  CVE: {CVE-ID}
  Severity: {severity_grade}
  ```
- Requires no additional infrastructure (no database, no network service, no external API)
- **Must not contain functional exploit code** — the vulnerability is introduced via the dependency version only

#### Image Tag

Specified in the Dockerfile as a comment at the top:

```dockerfile
# Image tag: claude-{language}-{CVE-ID}
```

For example:
```dockerfile
# Image tag: claude-python-CVE-2021-44228
```

## Step 7 — Validation Checklist

After generating all files for each language, verify:

- [ ] README.md exists in the language directory
- [ ] `{language}-set.csv` exists and has the correct number of rows (test cases + baseline rows)
- [ ] One directory exists per test case named by CVE ID
- [ ] One `base-{image}/` directory exists per base image in use
- [ ] Every Dockerfile has a pinned `FROM` (no `latest`)
- [ ] Every manifest file has all dependencies version-pinned
- [ ] No Dockerfile uses BuildKit-specific syntax
- [ ] Every `FROM` line includes `--platform=linux/amd64`
- [ ] No functional exploit code is present in any application file
- [ ] All CVSS scores traceable to NIST NVD

## Notes for Cross-LLM Comparability

The following must be consistent across all LLM-generated runs to preserve cross-LLM comparability:

- CSV field names and order are fixed as specified in Step 4.2 — do not add, remove or rename fields
- Directory naming convention must be followed exactly
- Image tag format must be followed exactly (substituting `claude` for the LLM name)
- CVSS scores sourced from NIST NVD only — no other source
- The same CVE search scope (March 2013 to present) applies to all runs

Divergence in CVE selection, base image choice, or Dockerfile implementation between LLM runs is academically significant and should not be artificially corrected.
