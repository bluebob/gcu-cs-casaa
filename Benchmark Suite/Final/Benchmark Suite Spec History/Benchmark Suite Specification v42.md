# Benchmark Suite Specification

Version 4.2

# Role

You are acting as a Senior DevOps Engineer assisting a SecOps engineer in their task of evaluating open source container vulnerability scanners. In order to this your colleague requires a benchmark suite to evaluate the scanners against.

Environment:

- you have full access to create directories and files as needed
- tasks should be run sequentially in the order given in this document. User confirmation is not required

## Tasks
### 1. Model specific
Create a subdirectory named for the LLM being used, for example gemini, claude, ChatGPT. Make that subdirectory the current directory. In that directory create a README.md detailing that this is the test suite generated by "xx" (whatever LLM is in use) for the degree research project.

### 2. Language scope

Core languages: Python, Go, Java and Javascript

Additional language requirement: C++ (included for completeness, replacing C). 

Selection criteria: languages have been selected based on the Tiobe long-term top 10 as per Feb 2026, excluding languages not native to Linux (Visual Basic, Delphi and C#) and application specific languages (specifically SQL). 

Each language is a 'test case'

### 3. Language test set
Carry out the the following steps for each language in scope. Process one language, then report results to user
#### 3.1 Test sets
Create a subdirectory of the language name and make that the current directory. These are referred to as "test sets"

Research the 12 most serious vulnerabilities with that language as specified in the NIST CVE database - either runtime or libraries.

The term 'most serious' should be based on CVSS3.1 (falling back to CVSS2 where version 3 is not available) scoring or other equivalent metrics, and should prefer "Critical" and "High" vulnerabilities, "Medium" may be used only if there are insufficient numbers of the more serious grades - under no circumstances should "Low" impact/grade vulnerabilities be used. 

The scope of the search should be from March 2013 (when Docker was first released) to now. If 12 is not possible then a lower limit of 10 samples may be used, but only if documented. If the limit is under 10 then the user should be informed and no further processing carried out, as this few samples is felt to give an insufficient scope of scanner results. The test set of 12 samples is being chosen to give a good balance between a useful set of scanner tests, while not being overly resource intensive to generate

#### 3.2 Verification and documentation

The LLM shall autonomously select the 12 most appropriate vulnerabilities based on the criteria. It must list these selections, including their CVSS scores and NVD links, in the language-specific `README.md` before generating the file structure. No user confirmation is required if the selection meets all specified criteria.

#### 3.3 Test cases
Repeat the steps for each vulnerability - these will be referred to as 'test cases'

Create a subdirectory with the name of the test case and make that subdirectory the current one

Create a Dockerfile to build a container with the test case with the following detailed specification:
- output should generate a viable container when run under 'docker build' or kaniko
- output shall consist of a Dockerfile plus language-standard manifest file - for example requirements.txt, go.mod or pom.xml. A suitable application may also be included
- resulant container should be runable, that is carry out a simple action without need of additional infrastructure, this shall be to print the name of the CVE, the effected package and the CVSS severity
- library packages and other dependencies shall be version pinned in the manifest file
- image tag should be LLM-language-CVE_number. If the vulnerability is named rather than being assigned a CVE then that name should be used instead of the CVE number
- there is no requirement for the LLM to build the containers, merely generate the files necessary for the container to subsequently be built 

A 'base' subdirectory should be created containing a Dockerfile to build a container using the chosen base image and nothing else other than a simple "hello world" application. This image will be used to provide a 'baseline' or reference for the results, allowing the removal of results due to vulnerabilities in the base image rather than the application layered on it. Where images other than the chosen reference base image are required - for example - to support a particular package or runtime vulnerability, then these should be created as subdirectories of the base directory and highlighted in the README.md in the test set directories.

#### 3.4 Documentation
The language directory shall contain two documentation items for the test case, these being generated BEFORE the test sets have been generated.

A README.md file should be created in the language subdirectory describing each vulnerability, description being the name of the vulnerability, the impact score and any relevant information, not to exceed two lines). A table should be given at the end of the document summarising the individual test cases in the test set. 

A CSV file should be created listing for each test case in the test: the CVE (or other description); CVSS score; affected library, runtime or package including version; the location of the test case; the base image in use. The file should be language-set.csv, for example python-set.csv, go-set.csv, etc. This file is designed to be processible by automation handling the test set (there is no requirement for a similar file for the test suite as a whole)

## Constraints

Under no circumstances should output be guessed, if a requirement cannot be satisfied then the user must be informed and asked for guidance. For example, the CVSS scores must be taken from the official databases only. Non canonical sources such as Reddit, Slashdot, Twitter/X and such should never be used.

The Dockerfile shall produce a viable container, and that container shall be storable in a Harbor registry. The container produced should be identical irrespective of whether kaniko or 'docker build' is used, and should avoid the use of any configuration incompatible with kaniko

Each container should -use the same base image, which should be a "slim" image, with alpine basis being strongly preferred. If the vulnerability in question is a language runtime one, or a package vulnerability that requires a specific non-standard base image (for example a vulnerability in a package that requires a python 3.3 base image), then a different base image can be chosen. If alpine is unsuitable, use the official language slim image (e.g., `python:3.12-slim`); document the deviation in the README and CSV. Where possible a particular image digest should be specified to exactly lock down the image being used - this being for reproducibility reasons. 

Target architecture should be linux/amd64. 

All containers should use the same version-specified base image (not 'latest')

## Exclusions

Mechanisms to build the containers and upload to the Harbor instances are not in-scope for this specification.

## Required structure

The example given below assumes that gemini was the LLM generating the benchmark suite. This is an example with only key items shown.

```
gemini
	+-- go
	+-- java
	+-- javascript
	+--- python
		+-- CVE-1234
			+-- Dockerfile
			+-- requirements.txt
			+-- app.py
		+-- CVE-4567
		+--README.md
		+-- base
			+-- base-alpine
			+--- base-python3.3
		+--python-set.csv
```
