# Benchmark Suite Specification

Version 5.5

## Overall Task

The task is to generate a benchmark suite of containers for a selection of programming languages. These containers will then be used to evaluate various open source container vulnerability scanners. An LLM (currently Claude, Gemini or ChatGPT) will be used to generate the files as defined in this specification

## Tasks
### 1. Model specific
Create a subdirectory named for the LLM being used, for example gemini, claude, ChatGPT. Make that subdirectory the current directory. In that directory create a README.md detailing that this is the test run generated by whatever LLM is in use for the degree research project.

### 2. Language scope
Scope of this is the programming languages Python, Go, Java, Javascript along with C++. Languages have been selected based on the Tiobe long-term top 10 as per Feb 2026, excluding languages not native to Linux (Visual Basic, Delphi and C#) and application specific languages (specifically SQL). C has not been included, since C++ has functionally replaced it.

Each language is a 'test set'

### 3. Language test set
Carry out the the following steps for each language in scope
#### 3.1 Test sets
Create a subdirectory of the language name and make that the current directory. These are referred to as "test sets"

Research the 12 most serious vulnerabilities with that language as specified in the NIST CVE database - either runtime or libraries.

The term 'most serious' should be based on CVSS3.1 (falling back to CVSS2 where version 3 is not available) scoring or other equivalent metrics, and should prefer "Critical" and "High" vulnerabilities, "Medium" may be used only if there are insufficient numbers of the more serious grades - under no circumstances should "Low" impact/grade vulnerabilities be used. 

The scope of the search should be from March 2013 (when Docker was first released) to now. If 12 is not possible then a lower limit of 10 samples may be used, but only if documented. If the limit is under 10 then the user should be informed and no further actions carried out, as this few samples is felt to give an insufficient scope of scanner results. The test set of 12 samples is being chosen to give a good balance between a useful set of scanner tests, while not being overly resource intensive to generate

#### 3.2 Test cases
Repeat the steps for each vulnerability - these will be referred to as 'test cases'

Create a subdirectory with the name of the test case and make that subdirectory the current one

The documentation items - as noted in the next section - should be generated first.

Create a Dockerfile to build a container with the test case with the following detailed specification:
- output should generate a viable container when run under 'docker build' or kaniko
- output shall consist of a Dockerfile plus language-standard manifest file - for example requirements.txt, go.mod or pom.xml. A suitable application in the language currently in scope should be included
- resultant container should be runnable, that is carry out a simple action without need of additional infrastructure, the recommendation is to print the name of the language, test case CVE and severity. There is no intention to collect or post-process this output.
- Dependencies should be loaded into the container by using the manifest file as specified in the previous item, and this shall be included in the Docker build process via one or more COPY commands in the Dockerfile
- library packages and other dependencies shall be version pinned in the manifest file
- image tag shall be LLM-language-CVE_number, where LLM is the name of the LLM generating the files. If the vulnerability is named rather than being assigned a CVE then that name should be used instead of the CVE number. For example claude-go-2026-4242
- there is no requirement for the LLM to build the containers, merely generate the files necessary for the container to subsequently be built 

#### 3.3 Documentation
The language directory shall contain two documentation items for the test cases in the test set, these being generated before the test cases themselves have been generated (as previously stated).

A README.md file should be created in the language subdirectory describing each vulnerability, description being the name of the vulnerability, the impact score and any relevant information (not to exceed two lines). A table should be given at the end of the document summarising the individual test cases in the test set. 

A CSV file should be created listing for each test case in the test: the CVE (or other description); CVSS score; affected library, runtime or package including version; the location of the test case; the base image in use. The file should be language-set.csv, for example python-set.csv, go-set.csv, etc. This file is designed to be processible by automation handling the test set (there is no requirement for a similar file for the test suite as a whole)

#### 3.4 Base image 

The base images used for the test suite may themselves contain vulnerabilities, which would adversely impact the scanner results To prevent this a 'base image' is required, and will be used to provide a 'baseline' or reference for the results, allowing the removal of results due to vulnerabilities in the base image rather than the application layered on it, or where a chosen package vulnerability requires a particular baseline image. Where images other than the chosen reference base image are required - for example - to support a particular package or runtime vulnerability, then these should be created as subdirectories of the test set directory (in the same manner as the test cases) and highlighted in the README.md in the test set directories.

These 'baseline case' containers should use the same base image, which should be a "slim" image, with alpine basis being strongly preferred. If the vulnerability in question is a language runtime one, or a package vulnerability that requires a specific non-standard base image (for example a vulnerability in a package that requires a python 3.3 base image), then a different base image can be chosen. If alpine is unsuitable, use the official language slim image (e.g., `python:3.12-slim`); document the deviation in the README and CSV. Where possible a particular image digest should be specified to exactly lock down the image being used - this being for reproducibility reasons. 

All containers should use the same version-specified base image (not 'latest')

A directory should be created containing a Dockerfile to build a container using the chosen base image and nothing else other than a simple application in the language under test, and will be named "base-" followed by the name of the image, for example base-alpine2.12. The simple application will print to screen the name of the language, and the "test case CVE and severity" will be replaced with the name of the base image and a severity of "baseline". If more than one base image is in use, then each base image should have a similarly named directory, e.g. base-python3-3.9. There is no intention to collect or post-process the output from the sample application.

Image tag will be LLM-language-directory_name, where directory_name is the name of the directory used for the baseline image, e.g. base-alpine2.12, so a gemini generated python baseline using Alpine 2.12 would be tagged gemini-python-base-alpine2.12




## Constraints

Under no circumstances should output be guessed, if a requirement cannot be satisfied then the user must be informed and asked for guidance. For example, the CVSS scores must be taken from the official databases only. Non canonical sources such as Reddit, Slashdot, Twitter/X and such should never be used.

The Dockerfile shall produce a viable container, and that container shall be storable in a Harbor registry. The container produced should be identical irrespective of whether kaniko or 'docker build' is used, and should avoid the use of any configuration incompatible with kaniko

Target architecture should be linux/amd64. 

## Exclusions

Mechanisms to build the containers and upload to the Harbor instances are not in-scope for this specification.

```
gemini
	+-- go
	+-- java
	+-- javascript
	+--- python
		+-- CVE-1234
			+-- Dockerfile
			+-- requirements.txt
			+-- app.py
		+-- CVE-4567
		+-- README.md
		+-- base-alpine2.12
		+-- base-python3.3
		+-- python-set.csv
```

```
		+-- CVE-4567 <-- "Test Case"
		+-- CVE-8900

	+-- python
```
